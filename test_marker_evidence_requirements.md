# 試験マーカー出力・エビデンス自動切り分けツール 要件定義書

## 概要
本書は、試験実施時にサービスの既存ログへマーカーを出力し、試験後にそのマーカーをもとに試験単位でエビデンスログを切り分けるための 2 つのプログラムについて要件を定義するものである。対象は、常時稼働しているアプリケーションや MQTT ブローカー、API サービスなどの既存ログを利用して試験エビデンスを取得する運用であり、試験ごとにログファイル自体を分けられない前提を置く[web:269][web:271][web:283]。

対象プログラムは次の 2 本とする。

- マーカー出力付き試験実行プログラム
- マーカーを用いたエビデンス自動抽出プログラム

## 背景
サービス常駐型のシステムでは、アプリケーションログやイベントログは継続的に追記されるため、試験単位でログファイルを分離する運用は難しい[web:271][web:278]。そのため、試験開始・終了や試験識別子をログ上で識別可能にし、後処理で必要範囲を抽出するログ管理の考え方が有効となる[web:269][web:273][web:283]。

また、ログ管理では「誰が、いつ、何を実施したか」を追跡できることが重要であり、時刻・実施者・対象試験を紐付けて記録できる設計が求められる[web:269][web:277][web:279]。

## 対象システム
| 項目 | 内容 |
|---|---|
| 対象ログ | アプリケーションログ、MQTT 関連ログ、API サービスログ、syslog、journalctl 経由で参照可能なサービスログ[web:271][web:278] |
| 実行環境 | Linux 系 OS を主対象とする。systemd 環境では journalctl 利用も考慮する[web:278][web:283] |
| 利用者 | 試験担当者、QA エンジニア、保守担当者 |
| 想定用途 | 手動試験、疎通試験、API 試験、MQTT 試験、障害再現試験 |

## 目的
本ツール群の目的は次のとおりである。

- 試験対象サービスの既存ログを利用しながら、試験単位のエビデンスを取得できるようにする。
- 試験担当者が試験開始・終了時刻を手作業で記録しなくても、ログ上のマーカーからエビデンスを抽出できるようにする。
- 1 つの大きなログファイルから、試験ごとの証跡ファイルを自動生成できるようにする。
- 試験実施日時、試験名、実施者、試験結果を追跡可能にすることで、ログ管理と証跡管理の効率を上げる[web:269][web:273]。

## スコープ
### 対象範囲
- 試験開始マーカー、試験終了マーカーの出力
- 一意な試験識別子の採番
- 識別子一覧の保存
- マーカーを用いた既存ログからの範囲抽出
- 試験単位のエビデンスファイル生成
- 基本的な実行結果表示と異常終了時の検知

### 対象外
- アプリケーション本体のログ出力仕様変更
- ログ保管基盤そのものの刷新
- GUI での試験管理画面の提供
- 高度なログ分析基盤や SIEM 連携
- 監査向けの完全な電子署名機能

## 前提条件
- 試験対象システムは既存のログ出力を行っていること。
- 試験担当者は試験実行端末または踏み台サーバー上でスクリプトを実行できること。
- logger コマンドまたは同等の仕組みでマーカーを記録できること。記録先が syslog/journald に限られる場合は、抽出対象ログもそれに合わせること[web:278][web:283]。
- 試験識別子をログ上で検索可能な文字列として出力できること。

## プログラム1要件
## マーカー出力付き試験実行プログラム

### 機能要件
| ID | 要件 |
|---|---|
| F1-01 | 試験開始時に一意な試験識別子を生成できること。 |
| F1-02 | 試験識別子は日時、プロセス ID、必要に応じて試験名を含められること。 |
| F1-03 | 試験開始マーカーをログへ出力できること。 |
| F1-04 | 試験終了マーカーをログへ出力できること。 |
| F1-05 | 試験名称、実施者、開始時刻をマーカーとともに出力できること。 |
| F1-06 | 試験内で実行するコマンド群を順次実行できること。 |
| F1-07 | コマンド実行結果の成否を標準出力または別ファイルへ記録できること。 |
| F1-08 | 生成した試験識別子を抽出用管理ファイルへ保存できること。 |
| F1-09 | 異常終了時でも終了マーカー相当の失敗情報を残せること。 |
| F1-10 | 実行完了後に、利用者へ試験識別子と次ステップを表示できること。 |

### 入力要件
- 試験名
- 実行対象コマンドまたは試験シナリオ名
- 実施者名（省略時は OS ユーザー名を使用）
- 出力先ログ種別（syslog / journald / 標準出力ログ等）
- 識別子管理ファイルの保存先

### 出力要件
- ログ内マーカー
- 試験識別子一覧ファイル
- 実行結果メッセージ
- 必要に応じた試験実行ログ

### 異常系要件
- logger 実行失敗時は標準エラーへ通知し、終了コードを非 0 とすること。
- 試験コマンド失敗時は失敗したコマンド名と時刻を出力すること。
- 終了マーカーが出せない場合でも、少なくとも開始マーカーと失敗時刻を管理ファイルへ残すこと。

## プログラム2要件
## マーカーを用いたエビデンス自動抽出プログラム

### 機能要件
| ID | 要件 |
|---|---|
| F2-01 | 識別子管理ファイルを読み込み、対象試験一覧を取得できること。 |
| F2-02 | 各試験識別子に対して、既存ログから該当ログを抽出できること。 |
| F2-03 | 抽出対象は、開始マーカーから終了マーカーまでの範囲を基本とすること。 |
| F2-04 | 終了マーカーがない場合は、開始マーカー以降の一定範囲または次マーカー直前までを抽出できること。 |
| F2-05 | 試験ごとに個別のエビデンスファイルを生成できること。 |
| F2-06 | 出力ファイル名に試験識別子を含められること。 |
| F2-07 | 抽出件数と出力先を一覧表示できること。 |
| F2-08 | 抽出失敗または対象なしの試験識別子をレポートできること。 |
| F2-09 | syslog 形式、アプリケーションログ形式、journalctl 出力形式のうち、事前定義した形式に対応できること。 |
| F2-10 | 全試験の抽出結果サマリを 1 ファイルに出力できること。 |

### 入力要件
- 識別子管理ファイル
- 対象ログファイルパス、または journalctl の参照条件
- エビデンス出力先ディレクトリ
- 抽出モード（厳密抽出 / 簡易抽出）

### 出力要件
- 試験別エビデンスログファイル
- 抽出結果サマリファイル
- 実行時コンソールメッセージ

### 異常系要件
- 識別子管理ファイルが存在しない場合は処理を中止すること。
- 対象ログへアクセスできない場合はエラーを表示すること。
- 開始マーカーのみ存在し終了マーカーがない場合は、その旨をサマリに記録すること。
- 同一識別子が複数箇所に存在する場合は、重複検出として警告を出すこと。

## 非機能要件
| 観点 | 要件 |
|---|---|
| 保守性 | スクリプトは bash など標準的な実行環境で読める構成とし、マーカー生成、試験実行、抽出処理を関数単位で分離すること。 |
| テスト容易性 | 単体でマーカー出力だけを検証できること、抽出処理だけをサンプルログで検証できること。 |
| 運用負荷 | 日常の試験担当者が 1〜2 コマンドで運用できること。ログ保存先や出力先は設定ファイルまたは環境変数で切り替え可能であること。 |
| 拡張性 | 新しいログ形式や試験種別を追加しやすいこと。識別子フォーマットを変更可能であること。 |
| 性能 | 数十 MB〜数百 MB 程度のログファイルに対して、業務利用上許容できる時間で抽出できること。 |

## データ要件
### 試験識別子形式
- 形式例: `TEST_YYYYMMDD_HHMMSS_PID`
- 必要条件: ログ上で一意に検索可能であること
- 推奨条件: 試験名略称を付与できること

### 管理ファイル項目
| 項目 | 必須 | 説明 |
|---|---|---|
| test_id | 必須 | 試験識別子 |
| test_name | 推奨 | 試験名称 |
| tester | 推奨 | 実施者 |
| start_time | 必須 | 試験開始時刻 |
| end_time | 任意 | 試験終了時刻 |
| status | 推奨 | 実行結果（OK/NG/ABORT） |
| evidence_file | 任意 | 抽出後の出力先 |

## 運用要件
### 基本運用フロー
1. 試験担当者がマーカー出力付き試験実行プログラムを起動する。
2. プログラムが試験開始マーカーと識別子を出力する。
3. 試験対象コマンドまたは操作を実施する。
4. プログラムが試験終了マーカーと結果を出力する。
5. 試験終了後、エビデンス自動抽出プログラムを実行する。
6. 試験別エビデンスファイルとサマリを確認する。

## セキュリティ・監査要件
- ログやエビデンスに機密情報が含まれる場合は、権限設定を適切に行うこと[web:273][web:277]。
- 識別子管理ファイルと生成されたエビデンスファイルには改ざん防止の観点から更新権限を限定すること[web:269][web:279]。
- 実施者情報を記録し、誰が試験を行ったか追跡できるようにすること[web:271][web:277]。

## リスクと対策
| リスク | 内容 | 対策 |
|---|---|---|
| マーカー未記録 | logger の失敗や権限不足により開始・終了マーカーが残らない | 実行前チェック、終了コード確認、管理ファイルへの退避記録 |
| ログ形式差異 | 環境によってログのフォーマットが異なる | ログ形式ごとの抽出関数を分離し、設定で切替可能にする |
| 同時試験衝突 | 複数試験が同時実行されログが混在する | 識別子に PID と時刻を含め、一意性を担保する |
| 異常終了 | 終了マーカー前にスクリプトが停止する | trap による異常終了記録を実装する |
| 誤抽出 | 同一文字列を他ログが含む | 識別子命名規則を厳格化し、前後トークン付きで検索する |

## 受入条件
- 試験実行プログラムを 1 回実行すると、開始マーカーと終了マーカーがログへ出力されること。
- 生成された試験識別子が管理ファイルに保存されること。
- 抽出プログラムを実行すると、試験識別子ごとに 1 ファイル以上のエビデンスが生成されること。
- 失敗試験または終了マーカー欠落時に、サマリへ警告が出力されること。
- 利用者が追加の手作業なしに、試験別エビデンスを参照できること。

## 推奨実装方針
実装は bash を第一候補とし、ログ抽出は grep、awk、sed、必要に応じて journalctl を利用する構成が現実的である。理由は、Linux 系環境で標準コマンドとして利用しやすく、既存の試験運用に組み込みやすいためである[web:278][web:283]。

一方で、ログ形式のばらつきや JSON 形式の管理ファイル、複数環境対応を強める場合は Python 実装を代替案として検討する価値がある。特に保守性とテスト容易性を重視する場合、抽出ロジックの単体試験は Python の方が書きやすい。

## 補足
- `logger` は syslog/journald にメッセージを書き込むためのコマンドであり、サービスログ本体に直接書き込めない環境では、抽出対象を syslog または journalctl 側に寄せる設計が必要となる[web:278][web:283]。
- 「マーカー」は試験境界を示す識別文字列であり、ログから対象区間を特定するためのタグを意味する。
